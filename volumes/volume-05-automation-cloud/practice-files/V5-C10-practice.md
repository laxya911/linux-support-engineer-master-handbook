# Practice Guide: Chapter 10 (Volume 5)

## Objective
To conceptually analyze the Golang (Go) structure required to define a custom Terraform Resource Schema and its CRUD functions.

## Assignment 1: The Resource Schema
When a user writes a `.tf` file, Terraform needs to know if the arguments they typed are valid. We define this in the Go `Schema`.

1. Review this theoretical Go code utilizing the `terraform-plugin-sdk/v2`:

    ```go
    package customapi

    import (
        "github.com/hashicorp/terraform-plugin-sdk/v2/helper/schema"
    )

    // resourceServer defines the schema for our custom "billing_account" resource
    func resourceServer() *schema.Resource {
        return &schema.Resource{
            
            // 1. Map the CRUD functions
            Create: resourceServerCreate,
            Read:   resourceServerRead,
            Update: resourceServerUpdate,
            Delete: resourceServerDelete,

            // 2. Define the Schema (What the user can type in the .tf file)
            Schema: map[string]*schema.Schema{
                "account_name": &schema.Schema{
                    Type:     schema.TypeString,
                    Required: true,   // The user MUST provide this
                },
                "spending_limit": &schema.Schema{
                    Type:     schema.TypeInt,
                    Optional: true,   // The user CAN provide this
                    Default:  1000,   // If they don't, it defaults to 1000
                },
                "api_token": &schema.Schema{
                    Type:      schema.TypeString,
                    Computed:  true,   // The API generates this! The user CANNOT type it.
                    Sensitive: true,   // Hides the token from the CLI output
                },
            },
        }
    }
    ```

2. **Analysis:** Based on this schema, if a user types `spending_limit = "One Thousand"` in their `.tf` file, Terraform Core will immediately throw an error before even contacting the API, because the Schema defines it strictly as `TypeInt`.

## Assignment 2: The Create Function
Let's look at a conceptual `Create` function.

1. Review the Go code:

    ```go
    func resourceServerCreate(d *schema.ResourceData, m interface{}) error {
        // 1. Extract the data the user typed in the .tf file
        name := d.Get("account_name").(string)
        limit := d.Get("spending_limit").(int)

        // 2. Initialize our theoretical API Client (passed via the 'm' interface)
        apiClient := m.(*MyInternalAPIClient)

        // 3. Make the HTTP POST Request to the API
        createdAccount, err := apiClient.CreateAccount(name, limit)
        if err != nil {
            return err // If the API fails, abort and show the user the error
        }

        // 4. CRITICAL STEP: Save the ID generated by the API into the State File
        d.SetId(createdAccount.ID)

        // 5. Save the generated API Token into the state file (the Computed field)
        d.Set("api_token", createdAccount.Token)

        // 6. Automatically run the Read function to ensure the state is perfectly synchronized
        return resourceServerRead(d, m)
    }
    ```

2. **Analysis:** The `d.SetId()` function is the most important line of code in the entire provider. Without it, Terraform cannot track the resource.

## Success Criteria
You have successfully completed this practice if you can explain what the `Computed: true` flag means in the Schema. (Answer: It tells Terraform that the user is not allowed to define this value in the `.tf` file. Instead, Terraform expects the API to generate this value during the `Create` phase, and the Provider will save it into the state file so other resources can reference it!).
